# data   Medallion Lake (Local)

Local on-disk mirror of the Fabric OneLake medallion structure. Used for DuckDB-based development without a Fabric connection.

---

## Stack

| Component | Technology |
|---|---|
| Bronze / Silver / Gold | Parquet (Hive-partitioned) |
| Query engine | DuckDB (local dev), Fabric Warehouse (production) |
| Cache | FastF1 HTTP cache (`cache/`) |

---

## Coverage

| Layer | Contents | Size |
|---|---|---|
| Bronze | 168 races × 4 datasets, 2018–2024 (laps, sectors, pits, telemetry)*; telemetry incomplete for 2018 Rd1/Rd2 | ~8 GB |
| Silver | 7 staging views + 4 reference tables, schema-tested |   |
| Gold | 5 feature mart tables (fct_driver_skill_features, fct_cliff_prediction_features, + dims) |   |

---

## Directory structure

```
data/
├── bronze/            # Raw FastF1 Parquet   Hive-partitioned by season/round
├── silver/            # Cleaned staging Parquet   dbt staging models materialised locally
├── gold/              # Feature mart Parquet   model-ready ML inputs
├── cache/             # FastF1 HTTP cache   avoids re-downloading telemetry
├── dev.duckdb         # Local development DuckDB (dbt dev target)
└── ci.duckdb          # CI DuckDB (dbt ci target)
```

---

## Contracts

This directory is consumed by:
- `transform/`   dbt reads Bronze and writes Silver + Gold
- `ml/`   reads Gold feature marts
- `app/`   reads Gold for app queries

**Bronze is append-only**   never modified after ingestion. All business logic lives in `transform/`.

---

## Quick start

```bash
make ingest-all     # populate Bronze
make dbt-dev-full   # populate Silver + Gold via dbt
```

## Known issues / open questions

- **2018 Rd1/Rd2 telemetry incomplete**   F1 didn't publish livetiming feed until later in 2018 season. FastF1 returns laps/weather/race_control but no telemetry for these 2 races. Accepted by design (ADR-009); ML layer handles via conditional feature selection.
- `dev.duckdb` is dbt's dev target; `ci.duckdb` is the CI target (see `transform/profiles/profiles.yml`)
- Silver Parquet is generated by `dbt run`   delete and re-run if schema changes

## Partition grammar

Every Bronze file follows this path structure-use it to decode any path you see:

```
data/bronze/<dataset>/season=<YYYY>/race=<event-slug>/[session=<Q|R>]/<file>.parquet
```

| Segment | Values | Notes |
|---|---|---|
| `<dataset>` | `laps`, `weather`, `race_control`, `telemetry` | One directory per dataset |
| `season=<YYYY>` | `2018` – `2024` | Hive-style partition key |
| `race=<event-slug>` | FastF1 `EventName` lowercased, spaces → hyphens | e.g. `bahrain-grand-prix`, `miami-grand-prix` |
| `session=<Q\|R>` | `Q` (qualifying), `R` (race) | **Telemetry only**-distinguishes session within round |
| `<file>.parquet` | Typically one file per partition | Written by `ingestion/src/ingest.py` |

**Exceptions:**
- 2018 Rd1/Rd2: telemetry Parquet absent (FastF1 livetiming feed started mid-season)
- `.gitkeep` files hold empty partition directories in version control
