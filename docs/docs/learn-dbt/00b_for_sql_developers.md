# SQL Developer Fast-Path

You're fluent in SQL. You've never used dbt. This page gives you everything you need to open any model in the pipeline and orient instantly in one read.

---

## The mental model

A dbt model is a single `SELECT` statement saved as a `.sql` file. When you run `dbt run`, dbt:

1. Reads every `.sql` file under `models/`
2. Parses `{{ ref('model_name') }}` calls to build a dependency graph (DAG)
3. Topologically sorts the DAG upstream models always build first
4. Materialises each model as a view or table in DuckDB

You never write `CREATE TABLE`, `DROP TABLE`, or manage run order manually. dbt handles all of it.

---

## Plain SQL ↔ dbt cheatsheet

| Plain SQL | dbt equivalent |
|---|---|
| `CREATE VIEW AS SELECT …` | A model file with `{{ config(materialized='view') }}` |
| `CREATE TABLE AS SELECT …` | A model file with `{{ config(materialized='table') }}` |
| Manual `DROP`/`CREATE` ordering | The DAG `{{ ref('x') }}` declares the dependency |
| Stored-proc parameters | dbt variables (`{{ var('bronze_base', '../data/bronze') }}`) and macros |
| Ad-hoc QA queries | Singular tests in `tests/` (SQL that returns rows = failure) |
| `INFORMATION_SCHEMA` column comments | `description:` fields in `schema.yml` → autogenerated dbt docs |
| Schema prefixes (`dev.`, `prod.`) | dbt targets (`--target dev`, `--target ci`) in `profiles/profiles.yml` |

## Read one model end-to-end

`int_lap_fuel_state.sql` is a good example: it's self-contained, well-commented, and shows the standard CTE chain pattern used throughout this pipeline.

```sql
{{ config(materialized='view') }}        -- ← dbt materialisation directive

WITH laps AS (                           -- ← source from upstream model
    SELECT * FROM {{ ref('stg_laps') }}  -- ← ref() = typed DAG dependency
),

circuits AS (
    SELECT * FROM {{ ref('dim_circuits') }}
),

joined AS (
    SELECT
        l.lap_id,
        l.race_year,
        l.race_id,
        l.driver_id,
        l.lap_number,
        -- Fuel load at start of lap (linear burnoff model)
        c.fuel_consumption_rate_kg_per_lap
            * (l.total_laps-l.lap_number + 1)        AS fuel_mass_kg,
        -- Weight correction: each kg of fuel = weight_penalty_factor seconds slower
        l.lap_time_s
           -c.weight_penalty_factor
            * c.fuel_consumption_rate_kg_per_lap
            * (l.total_laps-l.lap_number + 1)        AS weight_corrected_lap_time
    FROM laps l
    JOIN circuits c ON l.circuit_key = c.circuit_key
)

SELECT * FROM joined
```

**What to notice:**
- `{{ ref('stg_laps') }}` → dbt knows `stg_laps` must build first
- `{{ config(materialized='view') }}` → no table created, just a view
- No `CREATE`, no `DROP`, no schema prefix-dbt adds those at compile time
- The CTE chain (`laps →  circuits → joined → SELECT *`) is the standard pattern across all 36 intermediate models

## How columns are documented

Each model has a matching entry in its layer's `schema.yml`. Column descriptions there become autogenerated Reference pages (see the [Reference section](/reference/models/)). The Reference pages are generated from the dbt manifest by `scripts/build_reference.py` they stay in sync automatically. Do not hand-edit them.

```yaml
# transform/models/intermediate/schema.yml
- name: int_lap_fuel_state
  description: >
    Fuel burnoff and weight correction. One row per valid lap.
  columns:
   -name: lap_id
      description: Foreign key to stg_laps
      tests: [not_null, unique]
   -name: weight_corrected_lap_time
      description: Lap time with fuel weight penalty removed (seconds)
```

## The layer story in one paragraph

**Staging** cleans Bronze Parquet into typed, validity-flagged views. **Reference** provides seed-backed dimensions (circuits, compounds, drivers, constructors). **Intermediate** is where the physics happens across three sub-layers: L03 computes deterministic physical corrections (fuel, dirty air, thermal loads, wind, line deviation); L04 builds expected-pace baselines (compound cliff, constructor pace, driver ratings); L05 subtracts all corrections from actual lap times to produce the driver skill residual and event flags. **Marts** aggregate these outputs into feature tables for ML and analytics.
