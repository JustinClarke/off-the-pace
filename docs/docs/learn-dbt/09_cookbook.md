# Part 9: Common task recipes

This cookbook provides actionable, step-by-step recipes for completing common development and maintenance tasks within the transformation pipeline.

---

## Recipe 1  Running Specific Parts of the Lineage

To isolate development without rebuilding all 46 models, utilize dbt's powerful model selection syntax. Execute these commands from the `transform/` root directory:

```bash
# Compile and run a specific model
dbt run --profiles-dir profiles --select stg_laps

# Build a model and all of its upstream dependencies
dbt run --profiles-dir profiles --select +int_synthetic_teammate

# Build a model and all of its downstream dependents
dbt run --profiles-dir profiles --select int_lap_fuel_state+

# Build models in a specific folder
dbt run --profiles-dir profiles --select staging.*

# Run only the intermediate math identity tests
dbt test --profiles-dir profiles --select assert_lap_residual_identity
```

## Recipe 2  Integrating a New Racing Season

When a new racing season is completed and telemetry becomes available, integrate it into the pipeline using the following sequence:

1.  **Ingest Raw Data:** Run the Python ingestion script to retrieve telemetry, lap data, weather, and pit stops from FastF1:
    ```bash
    python ingestion/src/ingest.py --start-season 2025 --end-season 2025 --sessions both
    ```
2.  **Build Database:** Recompile and build the DuckDB schemas. The wildcard glob path in `src_formula1.yml` automatically picks up the new season directories:
    ```bash
    make dbt-dev
    ```
3.  **Run Quality Checks:** Verify that the new data does not violate schema contracts or relationship constraints:
    ```bash
    make dbt-test
    ```
4.  **Refit wear models:** (See Recipe 3 below) to calibrate compound parameters on the expanded historical window.

## Recipe 3  Re-fitting Compound Cliff Coefficients

Tyre compounds and wear rates change with rule updates. To re-fit the Kaplan-Meier survival curves on the active data window, run the offline survival fitter:

1.  **Run survival fitter:** Execute the Python coefficient task:
    ```bash
    # Fits compound cliff wear models and circuit weight factors
    make coefficients-fit
    ```
    This generates new `.csv` parameter sheets and places them in `seeds/_pending/`.
2.  **Verify pending fit:** Review the parameters inside `seeds/_pending/compound_cliff_params.csv` and verify that the fitted onset values ($\tau$) are physically plausible.
3.  **Promote parameters:** Promote the pending seeds to active seeds:
    ```bash
    make coefficients-promote
    ```
    This moves the pending files to `seeds/` and archives previous versions in `seeds/_archive/`.
4.  **Rebuild pipeline:** Re-run the dbt seeds and models to compile the newly promoted coefficients:
    ```bash
    make dbt-dev-full
    ```

## Recipe 4  Debugging Mathematical Identity Violations

If an upstream refactor alters a timing calculation, a singular validation test (like `assert_lap_residual_identity`) will fail. To locate the specific rows causing the violation:

1.  **Locate Compiled Query:** Open `/transform/target/compiled/off_the_pace/tests/assert_lap_residual_identity.sql`. This is the raw SQL generated by the `assert_additive_identity` macro.
2.  **Run Diagnostic Query:** Copy that SQL query, append a projection clause, and execute it against `data/dev.duckdb` using Harlequin (`make query`):
    ```sql
    -- Diagnostic check for identity delta
    SELECT 
        lap_id, 
        driver_id,
        pace_delta_s,
        (fuel_component_s + compound_component_s + rubber_component_s +
         ambient_component_s + constructor_component_s + dirty_air_tax_s) AS sum_components,
        driver_skill_residual_s,
        ABS(pace_delta_s-(
            fuel_component_s + compound_component_s + rubber_component_s +
            ambient_component_s + constructor_component_s + dirty_air_tax_s +
            driver_skill_residual_s
        )) AS identity_violation_delta
    FROM int_lap_residual_decomposed
    WHERE identity_violation_delta > 0.0001
    LIMIT 10;
    ```
3.  **Analyze Violations:** If the output shows a static delta across multiple rows, check which intermediate component was recently modified and confirm that its timing variable is correctly integrated into `int_lap_residual_decomposed.sql`.
