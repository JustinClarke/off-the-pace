# Reproducibility & deployment

A model nobody can reproduce is a screenshot. The machine layer is built so that the same data and the same command produce **byte-identical** artefacts, and so those artefacts run unchanged in a browser. This page covers the determinism proof and the ONNX deployment path.

---

## Determinism proven, not assumed

Two things are pinned and verified:

| Check | Result |
|---|---|
| Dataset fingerprint, two independent loads | `3aff4559…` **identical both times** |
| `degradation_regressor_p50_v1.bst` SHA256, before vs after a fresh retrain (same params, same seed, same data) | `2031a408…` **byte-identical across sessions** |

The chain is: `features.py` turns the mart into a deterministic `float32` matrix and emits a **SHA256 fingerprint**; a fixed `RANDOM_STATE = 20260528`, the persisted Optuna study, and pinned library versions reproduce the exact booster in a fresh process. The other four boosters ride the identical machinery. Provenance recorded in the [model card](/reference/ml/degradation-model-v1):

- **random_state:** `20260528`
- **dataset fingerprint:** `3aff45592c335b97…`
- **library versions:** `xgboost 3.2.0`, `scikit-learn 1.8.0`
- **command:** `make ml-all`

## ONNX export for in-browser scoring

Every booster is exported to **ONNX** and gated by a parity test: each `.bst` must round-trip to its `.onnx` within `atol = 1e-5` on a **NaN-bearing** sample. All five pass, with maximum absolute differences in the **1e-6 to 1e-5** range and crucially the quantile (`reg:quantileerror`) models round-trip faithfully, which was the single biggest technical risk of the whole layer (resolved early, during the ONNX parity spike milestone).

The React + DuckDB-Wasm app will score laps client-side from the `.onnx` files sub-10ms inference, zero server compute. The browser needs no Python and no warehouse.

## The publish manifest

`export_onnx.py` finalises `ml/models/manifest.json` on a full all-pass export the single index the browser reads to score the models *correctly*:

- the **input contract** (positional 38-feature order, inlined `encoders.json`, the `-1` missing-categorical sentinel, the native-NaN note),
- per-model **output interpretation** (quantile bounds `[−10, 10]`; classifier ZipMap with class order; stint-life `clip(≥0)`),
- the 17-column predictions schema, and
- **provenance** (training seasons, holdout, fingerprint, library versions, parity results).

It is tracked in git but **never copied to `app/`**. The small metadata files (`encoders.json`, `manifest.json`, `model_card.json`) are tracked; the large binaries (`.bst`, `.onnx`, parquet, Optuna DBs, PNGs) are git-ignored and regenerated by `make ml-all`.

## Predictions output

`predict.py` scores **all 137,373 laps** (training and holdout, eligible or not) into `data/marts/mart_degradation_predictions.parquet` against a fixed 17-column Arrow schema schema drift aborts the write. That parquet is the analytics-ready output the app and notebooks consume.

## Continuous integration

`.github/workflows/ml-ci.yml` mirrors the dbt CI pattern: a **leakage-guard** job runs the static guards always and the warehouse guards when bronze fixtures are present, and an **ml-smoke** job rebuilds the mart, runs a tiny train, exports ONNX and runs the parity/predict tests. The reference model card is regenerated by `scripts/gen_ml_reference.py` and a `--check` flag fails CI on any drift between `ml/model_card.yml` and the published MDX the docs cannot go stale-green.
